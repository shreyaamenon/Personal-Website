Overfitting always came across as a vague buzzword to me at first. I assumed it just meant the model was extremely accurate, which I thought was the entire goal.

But in class, I learned it’s more nuanced than that. An overfit model is like a student who memorizes every single line of the textbook but struggles to answer questions that are even slightly different from the examples they studied. The model essentially memorizes the training data instead of learning the patterns behind it, which makes it perform poorly when it encounters anything new.

The main ways to address overfitting are actually straightforward: provide the model with more diverse data, simplify its structure so it can’t memorize every detail, or apply regularization techniques to control complexity. The goal is to help the model understand the underlying concepts in the data — not just store every tiny detail it has seen before.

Data leakage happens when information from outside the training dataset sneaks into the model in a way that should not happen. This can make the model look far more accurate during training and validation than it really is. For example, if a model predicting credit risk accidentally has access to a column that directly indicates whether someone defaulted, it will “cheat” and use that instead of learning real patterns.

While working on a small project, I realized how easy it is to overlook leakage. Even simple mistakes like splitting data after preprocessing can let information leak between the training and test sets. It became clear that preventing leakage means carefully reviewing data sources, the order of processing steps, and which features are used, so the model only learns from information it would realistically have in the real world.

Precision and recall were two terms I mixed up for a long time. Precision is about how many of the model’s positive predictions were actually correct, while recall measures how many of the true positives the model successfully found. These metrics are especially important when false positives or false negatives have very different consequences.

I remember the difference by thinking of precision as “how precise is the model’s positive predictions?” and recall as “how well does the model recall the true positives?” This helps me keep them straight in my mind. While experimenting with a classification project, I found that focusing on only accuracy could hide these details. A model might have been “accurate,” but it was missing important cases or flagging too many incorrect ones. Looking at precision and recall together finally made it easier to understand the trade-offs and choose the right balance for the problem.

Regularization sounded complicated when I first heard it mentioned in class. At its core, it is a technique used to prevent overfitting by adding a penalty to overly complex models. This penalty discourages the model from relying too heavily on any one feature or memorizing small quirks in the training data.

I used to think of regularization as just another math trick, but it clicked when I realized it’s about keeping the model focused on the most important patterns. It’s like a coach telling an athlete to avoid unnecessary movements and focus on the fundamentals. In practice, I found that using L1 or L2 regularization can help the model generalize better by keeping it from getting distracted by noise in the data. So, in one of my projects, using L1 and L2 regularization finally made sense. L1 tends to push some feature weights to zero, essentially performing feature selection, while L2 shrinks weights more evenly. It showed me how regularization can keep a model focused on the general patterns rather than every minor fluctuation in the data.

When I first started working with datasets, I sometimes used to split the data into train and test sets after normalizing or encoding it. I later learned that this can create problems, because it allows information from the test set to influence how the training data is processed.

The right way to do it is to split the data first, then apply any transformations separately to the training and test sets. This way, the model only learns from the training data and is evaluated on truly unseen data. I learned this the hard way when I noticed that my model performed well on the test set, but when I tried it on new data, it didn’t do as well. It turned out that the test data had influenced the preprocessing steps, leading to overly optimistic results. In a later project, I finally got into the habit of splitting the data before doing any transformations. This way, the test data truly stays unseen until evaluation, and the model’s performance is a fair reflection of how it will handle real-world data it has never seen.
